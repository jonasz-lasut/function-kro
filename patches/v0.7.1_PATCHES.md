# Function-KRO Adaptations for KRO v0.7.1

This document describes the adaptations made to upstream KRO v0.7.1 code to work as a Crossplane composition function. An AI agent should use this document to understand the **intent** behind each change and apply similar adaptations when upgrading to newer KRO versions.

## Core Problem

Upstream KRO runs as a standalone Kubernetes controller that:
1. Gets OpenAPI schemas by querying the Kubernetes API server directly
2. Has full access to a `rest.Config` for discovery and REST mapping
3. Creates resources directly via dynamic client

Function-KRO runs as a Crossplane composition function that:
1. Cannot query the API server - must get schemas via Crossplane's `required_schemas` or `required_resources` mechanisms
2. Has no `rest.Config` - must work without discovery or fall back to conventions
3. Returns desired resources to Crossplane rather than creating them directly

The adaptations below bridge this gap.

---

## Adaptation 1: Schema Resolution Strategy

### Problem
Upstream KRO's `graph.Builder` expects to receive CRDs or query schemas from a live API server. Function-KRO receives schemas from Crossplane via the function protocol.

### Solution
We support **two schema resolution paths** for maximum compatibility:

1. **Primary: `required_schemas`** - Modern Crossplane versions provide OpenAPI schemas directly
2. **Fallback: `required_resources`** - Older Crossplane versions require fetching CRDs and extracting schemas

### Files Added

**`kro/graph/schema/resolver/schema_map_resolver.go`** (new file)
- `SchemaMapResolver` - holds a map of GVK to `*spec.Schema`
- `StructToSpecSchema()` - converts Crossplane's protobuf Struct to `spec.Schema`
- `combinedResolver` - chains multiple resolvers with fallback behavior

**`kro/graph/schema/resolver/crd_resolver.go`** (new file)
- `CRDSchemaResolver` - extracts schemas from CRD objects for the fallback path
- Uses `ConvertJSONSchemaPropsToSpecSchema()` to convert CRD validation schemas

### Files Modified

**`kro/graph/schema/resolver/resolver.go`**
- Added `NewCombinedResolverFromSchemas()` - creates resolver from schema map + core types
- Added `NewCombinedResolverFromCRDs()` - creates resolver from CRDs + core types
- Added `newCoreResolver()` - resolves built-in K8s types (Deployment, Service, etc.)

### Key Insight for Future Versions
The resolver abstraction (`resolver.SchemaResolver` interface) isolates the rest of KRO from how schemas are obtained. When upgrading, preserve this abstraction and ensure the new KRO code uses it rather than assuming API server access.

## Adaptation 2: Builder Constructor Changes

### Problem
Upstream KRO's `NewBuilder()` accepts CRDs directly and builds internal state (GVK-to-GVR mappings, namespace scope, etc.) from them. Function-KRO needs to accept a pre-built resolver.

### Solution

**Changed `kro/graph/builder.go`:**

```go
// Upstream signature (conceptual):
func NewBuilder(crds ...*extv1.CustomResourceDefinition) (*Builder, error)

// Function-KRO signature:
func NewBuilder(schemaResolver resolver.SchemaResolver, restMapper meta.RESTMapper) *Builder
```

The `restMapper` parameter is optional (can be nil). When nil, we use a convention-based fallback:
- GVR = `{group}/{version}/{lowercase(kind)}s`
- Assume namespaced = true

### Key Insight for Future Versions
If upstream KRO changes the Builder constructor, preserve the pattern of accepting an external resolver rather than building one internally. The resolver should be constructed in `fn.go` where we have access to Crossplane's schema data.

## Adaptation 3: Graph Definition from Schema (not CRD)

### Problem
Upstream `NewResourceGraphDefinition()` accepts an XR CRD to extract the composite resource schema. Function-KRO receives the schema directly from Crossplane.

### Solution

**Changed `kro/graph/builder.go`:**

```go
// Upstream signature (conceptual):
func (b *Builder) NewResourceGraphDefinition(rg *v1beta1.ResourceGraph, xrCRD *extv1.CustomResourceDefinition) (*Graph, error)

// Function-KRO signature:
func (b *Builder) NewResourceGraphDefinition(rg *v1beta1.ResourceGraph, xrSchema *spec.Schema) (*Graph, error)
```

Similarly, `buildInstanceResource()` was changed to accept `*spec.Schema` instead of extracting from CRD.

### Key Insight for Future Versions
The XR schema flows through: Crossplane -> fn.go -> graph.Builder -> Graph. Ensure this data flow is preserved. If upstream adds new parameters, determine whether they can be derived from the schema or need to come from fn.go.

## Adaptation 4: REST Mapping Fallback

### Problem
Upstream KRO uses discovery to map GVK -> GVR and determine namespace scope. Function-KRO has no discovery client.

### Solution

**In `kro/graph/builder.go`, `buildRGResource()`:**

```go
var gvr = gvk.GroupVersion().WithResource(strings.ToLower(gvk.Kind) + "s")
var namespaced = true
if b.restMapper != nil {
    mapping, err := b.restMapper.RESTMapping(gvk.GroupKind(), gvk.Version)
    if err != nil {
        return nil, fmt.Errorf("failed to get REST mapping for resource %s: %w", rgResource.ID, err)
    }
    gvr = mapping.Resource
    namespaced = mapping.Scope.Name() == meta.RESTScopeNameNamespace
}
```

The convention-based fallback (pluralize kind, assume namespaced) works for most resources. We pass `nil` for restMapper from fn.go.

### Key Insight for Future Versions
If upstream KRO adds more sophisticated resource discovery, we may need to either:
1. Extend the fallback conventions
2. Have Crossplane provide this information
3. Accept slightly degraded functionality for edge cases

## Adaptation 5: ObjectMeta Schema Injection

### Problem
Crossplane's schemas include a `metadata` field, but it's an **unresolved `$ref`** pointing to ObjectMeta. KRO's CEL parser fails with:
```
schema at path metadata has no valid type, OneOf, AnyOf, or AdditionalProperties
```

### Solution

**In `kro/graph/schema/schema.go`:**
- Added `ObjectMetaSchema` variable - fully resolved ObjectMeta schema loaded at init time
- Added `DeepCopySchema()` helper for safe schema modification

**In `fn.go` and `crd_resolver.go`:**
Always inject the full ObjectMeta schema after receiving/extracting a schema:
```go
if specSchema.Properties == nil {
    specSchema.Properties = make(map[string]spec.Schema)
}
specSchema.Properties["metadata"] = kroschema.ObjectMetaSchema
```

### Key Insight for Future Versions
This is a Crossplane limitation, not a KRO issue. The injection must happen wherever schemas enter the system (both schema path and CRD path). If Crossplane starts resolving `$ref` before sending schemas, this injection becomes unnecessary but harmless.

## Adaptation 6: Dual-Path Schema Fetching in fn.go

### Problem
Different Crossplane versions support different mechanisms for providing schemas to functions.

### Solution

**In `fn.go`:**

```go
// Request BOTH schemas and CRDs - Crossplane will provide what it supports
rsp.Requirements = &fnv1.Requirements{
    Schemas:   RequiredSchemas(gvks...).Schemas,
    Resources: RequiredCRDs(gvks...).Resources,
}

// Try schemas first (preferred)
combinedResolver, xrSchema, err := f.trySchemaPath(req, gvks, xrGVK)
if err != nil {
    response.Fatal(rsp, err)
    return rsp, nil
}

// Fall back to CRDs if schemas unavailable
if combinedResolver == nil {
    combinedResolver, xrSchema, err = f.tryCRDPath(req, gvks, xrGVK)
    // ...
}
```

**Helper functions:**
- `RequiredSchemas()` - builds `fnv1.Requirements` with schema selectors
- `RequiredCRDs()` - builds `fnv1.Requirements` with CRD resource selectors
- `trySchemaPath()` - processes `req.GetRequiredSchemas()`
- `tryCRDPath()` - processes `req.GetRequiredResources()` as CRDs

### Key Insight for Future Versions
The dual-path approach adds complexity but ensures broad compatibility. When upgrading:
1. Keep both paths unless dropping support for older Crossplane versions
2. The schema path is preferred and should be tried first
3. Both paths must inject ObjectMeta and construct the same resolver interface

## Adaptation 7: SSA-Compatible Desired State

### Problem
KRO's runtime returns full resource objects with all fields. If we return these as desired state, Crossplane claims SSA ownership of every field, including provider-defaulted values.

### Solution

**In `fn.go`:**

Use `GetRenderedResource()` instead of `GetResource()` to get only the template-defined fields:
```go
r, state := rt.GetRenderedResource(id)
if state != runtime.ResourceStateResolved {
    continue
}
```

For the XR status, build a minimal desired XR containing only the status paths declared in the ResourceGraph:
```go
dxr := &composite.Unstructured{...}
for _, v := range g.Instance.GetVariables() {
    val, err := src.GetValue(v.Path)
    if err != nil {
        continue // Not resolved yet
    }
    dst.SetValue(v.Path, val)
}
```

### Key Insight for Future Versions
This is about Crossplane SSA semantics, not KRO internals. The runtime's `GetRenderedResource()` method is key - if it's renamed or the API changes, find the equivalent that returns template-only fields.

## Checklist for Upgrading to New KRO Version

When upgrading to a new upstream KRO version:

1. [ ] Copy new `pkg/graph/`, `pkg/cel/`, `pkg/runtime/`, `pkg/metadata/` from upstream to `kro/`
2. [ ] Check if `graph.Builder` constructor changed - adapt to accept resolver + optional restMapper
3. [ ] Check if `NewResourceGraphDefinition` signature changed - adapt to accept `*spec.Schema`
4. [ ] Check if runtime APIs changed - ensure `GetRenderedResource()` or equivalent exists
5. [ ] Verify `resolver.SchemaResolver` interface is still compatible
6. [ ] Re-add ObjectMeta injection in schema resolution paths
7. [ ] Run `go build ./...` and fix compilation errors
8. [ ] Run `go test ./...` and fix test failures
9. [ ] Update this document with any new adaptations

---

## Non-Goals / Intentionally Not Adapted

- **SimpleSchema processing** - Function-KRO receives pre-built XR schemas from Crossplane, so SimpleSchema-to-CRD generation is not needed
- **Controller/reconciler logic** - Function-KRO returns desired state, doesn't reconcile
- **CRD generation** - Crossplane handles XRD-to-CRD, not function-kro
- **Breaking schema change detection** - This is a controller-side feature for CRD updates
