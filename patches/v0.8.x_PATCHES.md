# Function-KRO Adaptations for KRO v0.8.x

This document describes the adaptations made to upstream KRO v0.8.3 code to make it work as a Crossplane composition function.

## Summary of Changes

| File | Change Type | Description |
|------|-------------|-------------|
| `kro/graph/builder.go` | Major Rewrite | Adapted for Crossplane function model |
| `kro/graph/schema/resolver/resolver.go` | Addition | Factory functions for schema resolvers |
| `kro/graph/schema/schema.go` | Addition | DeepCopySchema helper |
| `input/v1beta1/input.go` | Addition | KRODomainName constant, ForEach support |
| `kro/metadata/*.go` | Import Fix | Changed v1alpha1 to v1beta1 |
| `fn.go` | Major Rewrite | Updated for new runtime API |

## Detailed Adaptations

### 1. Builder Adaptations (`kro/graph/builder.go`)

**Intent:** Make the builder work with Crossplane's schema resolution instead of Kubernetes API server discovery.

**Changes:**

1. **Constructor signature:**
   ```go
   // Upstream (v0.8.x):
   func NewBuilder(clientConfig *rest.Config, httpClient *http.Client) *Builder

   // Function-kro:
   func NewBuilder(schemaResolver resolver.SchemaResolver, restMapper meta.RESTMapper) *Builder
   ```
   - Accepts pre-configured schema resolver (Crossplane provides schemas via required_schemas or CRDs)
   - RESTMapper is optional (nil triggers convention-based fallback)

2. **NewResourceGraphDefinition signature:**
   ```go
   // Upstream (v0.8.x):
   func (b *Builder) NewResourceGraphDefinition(rgd *v1alpha1.ResourceGraphDefinition) (*Graph, error)

   // Function-kro:
   func (b *Builder) NewResourceGraphDefinition(rg *v1beta1.ResourceGraph, xrSchema *spec.Schema) (*Graph, error)
   ```
   - Accepts our `ResourceGraph` input type (not the full RGD CR)
   - XR schema passed explicitly (Crossplane resolves it, not the builder)
   - No SimpleSchema processing (Crossplane provides OpenAPI schemas directly)

3. **REST mapping fallback:**
   When `restMapper` is nil, the builder uses convention-based GVR derivation:
   ```go
   var gvr = gvk.GroupVersion().WithResource(strings.ToLower(gvk.Kind) + "s")
   var namespaced = true
   ```

4. **Instance node dependencies:**
   Fixed bug where instance status expression dependencies weren't being set on the instance node's Meta.Dependencies field.

### 2. Schema Resolver Factories (`kro/graph/schema/resolver/resolver.go`)

**Intent:** Provide factory functions to create combined resolvers for different schema resolution paths.

**Additions:**

```go
// Create resolver from Crossplane's required_schemas
func NewCombinedResolverFromSchemas(schemaMapResolver *SchemaMapResolver) resolver.SchemaResolver

// Create resolver from CRDs (fallback path)
func NewCombinedResolverFromCRDs(crds []*extv1.CustomResourceDefinition) (resolver.SchemaResolver, error)
```

Both functions create a combined resolver that:
1. Tries the provided resolver first (schema map or CRD)
2. Falls back to core types resolver for built-in Kubernetes types

### 3. Schema Utilities (`kro/graph/schema/schema.go`)

**Addition:** `DeepCopySchema` function for safely copying OpenAPI schemas:
```go
func DeepCopySchema(schema *spec.Schema) *spec.Schema
```

### 4. Input Types (`input/v1beta1/input.go`)

**Additions:**

1. **KRODomainName constant:**
   ```go
   const KRODomainName = "kro.run"
   ```
   Used by metadata package for labels and finalizers.

2. **GroupVersion variable:**
   ```go
   var GroupVersion = schema.GroupVersion{Group: "kro.fn.crossplane.io", Version: "v1beta1"}
   ```
   Used by metadata package for owner references.

3. **ForEach support:**
   ```go
   type ForEachDimension map[string]string

   type Resource struct {
       // ... existing fields ...
       ForEach []ForEachDimension `json:"forEach,omitempty"`
   }
   ```

### 5. Metadata Package Changes

All files in `kro/metadata/` were updated to import from `input/v1beta1` instead of the upstream `api/v1alpha1`:
- `finalizers.go`
- `groupversion.go`
- `labels.go`
- `owner_reference.go`

### 6. Function Entry Point (`fn.go`)

**Major rewrite** to use the new v0.8.x runtime API:

**Old API (v0.7.x):**
```go
rt, err := g.NewGraphRuntime(&oxr.Resource.Unstructured)
rt.SetResource(id, resource)
rt.IsResourceReady(id)
r, state := rt.GetRenderedResource(id)
rt.TopologicalOrder()
rt.Synchronize()
```

**New API (v0.8.x):**
```go
rt, err := runtime.FromGraph(g, &oxr.Resource.Unstructured)
node.SetObserved([]*unstructured.Unstructured{resource})
node.IsReady()
node.GetDesired()  // Returns []*unstructured.Unstructured
rt.Nodes()         // Returns nodes in topological order
rt.Instance()      // Returns the instance node
```

Key changes:
- Runtime created via `runtime.FromGraph()` instead of graph method
- Node-based API: each resource is a `*runtime.Node`
- `GetDesired()` returns slices (for collection support)
- `IsIgnored()` replaces `ReadyToProcessResource()` for includeWhen evaluation
- `ErrDataPending` indicates CEL expressions waiting for data

## New Features Integrated

### Collections (forEach)

v0.8.x added support for expanding a single resource template into multiple resources using `forEach` dimensions. This is now supported:

1. Input type has `ForEach []ForEachDimension` field
2. Builder parses forEach dimensions and creates `NodeTypeCollection` nodes
3. Runtime's `node.GetDesired()` returns multiple resources for collections
4. fn.go handles collections by creating uniquely-named composed resources

## Schema Resolution Paths

Function-kro supports two schema resolution paths for maximum Crossplane compatibility:

### Path 1: Required Schemas (Preferred)
```
Crossplane → required_schemas → SchemaMapResolver → Combined Resolver
```
- Modern Crossplane versions support this
- Most efficient - schemas provided directly

### Path 2: CRDs (Fallback)
```
Crossplane → required_resources (CRDs) → CRDSchemaResolver → Combined Resolver
```
- Works with older Crossplane versions
- Extracts schemas from CRD objects

Both paths inject ObjectMeta schema and use the same core types resolver for built-in types.

## Known Limitations

1. **golangci-lint config:** The `.golangci.yml` needs updates for golangci-lint v2.x format (separate maintenance task)

2. **Collection resource naming:** Collections currently use index-based naming (`resourceId-0`, `resourceId-1`). A more robust naming scheme may be needed.

3. **External refs in collections:** Not fully tested with forEach expansion.
