# Function-KRO Adaptations for KRO v0.9.0

This document is the **upgrade guide** for re-applying function-kro's adaptations when upgrading to a new upstream KRO version. It is designed to be used by an agent following `UPGRADE_PROCESS.md` Phase 3.

For each adaptation, this document provides:
1. **What to look for** in the new upstream code
2. **What to do** (remove, replace, or modify)
3. **The complete function-kro code** where applicable, so you can reconstruct it even if upstream changed significantly
4. **Conditional guidance** for when upstream may have changed

Based on a full audit using `./scripts/diff-upstream-kro.sh -r v0.9.0`.

### Applying Mappings Globally

**IMPORTANT:** The Type Mapping and Field Access Mapping tables above must be applied **globally** across ALL functions in every modified file, not just the specific functions called out in the numbered adaptations below. When you encounter any reference to an upstream type or field path in any function, apply the appropriate mapping. The numbered adaptations below highlight the major structural changes, but simple type/field substitutions occur throughout.

### Guidance for New Upstream Fields

If a new upstream KRO version adds fields to `v1alpha1.ResourceGraphDefinition` or `v1alpha1.Resource`, you must decide how to handle them in `input/v1beta1/input.go`:

1. **Fields that map to Crossplane concepts** (e.g., schema, GVR): Skip -- Crossplane provides these via the function protocol
2. **Fields that affect resource behavior** (e.g., new options on Resources like readyWhen, includeWhen, forEach): Add the equivalent field to `input/v1beta1/input.go` `Resource` struct
3. **Fields on Schema** (e.g., new schema options): Evaluate -- function-kro receives the XR schema as a `*spec.Schema` parameter, so Schema-level fields are typically not needed
4. **Fields on ResourceGraphDefinition.Spec**: Evaluate -- our `ResourceGraph` has a flat structure (no `Spec` wrapper), so add directly to `ResourceGraph` if needed

After adding new fields, update the Field Access Mapping table above and ensure `builder.go` accesses them correctly.

## Changes from v0.8.x Adaptations

This section summarizes what changed in v0.9.0 compared to the v0.8.x adaptation set, to help orient the reader.

### New Upstream Concepts in v0.9.0

- **`buildContext` type**: Upstream added a `buildContext` struct (`bc`) that carries the CEL environment, type provider, and compilation/parse-check caches. Our `buildInstanceNodeFromSchema` now receives `bc` as a parameter instead of creating its own CEL environment.
- **`Expression` type**: CEL expressions are now wrapped in a `krocel.Expression` struct with `Original`, `UserExpression()`, and `References` fields. The parser returns `variable.FieldDescriptor` structs containing `Expression` objects rather than raw strings.
- **`RGDConfig` parameter**: `NewResourceGraphDefinition` and validation functions now accept an `RGDConfig` struct with max collection dimension size. We pass this through.
- **Feature gates**: Upstream added `sigs.k8s.io/kro/pkg/features` for `--feature-gates`. We vendor this package identically.
- **`NodeTypeExternalCollection`**: Upstream added a new node type for external refs with selectors. We pass it through unmodified (runtime handles it).
- **`CheckReadiness()` replaces `IsReady()`**: Readiness checking was refactored with error sentinels. No adaptation needed -- we use it as-is.
- **`Parser` is now a struct**: `parser.New(schemaCache)` returns a `*Parser` instance rather than the old package-level functions. No adaptation needed.
- **`inspectExpressionRestricted` + `bc.compile`**: Status parsing now uses the shared `buildContext` for expression inspection and compilation. Our `buildInstanceNodeFromSchema` uses these.
- **`node_resolve.go`**: `normalizeNamespaces` was moved from `node.go` to `node_resolve.go` in upstream. We still remove it from its new location.

### Adaptation Changes from v0.8.x

- **Adaptation 3 (buildInstanceNode)**: Renamed to `buildInstanceNodeFromSchema`. Now receives `bc *buildContext` instead of creating its own CEL environment. Uses `inspectExpressionRestricted` and `bc.compile` from the upstream `buildContext` API.
- **Adaptation 5 (buildRGResource)**: The `instanceNamespaced` parameter was removed from this function's signature in upstream v0.9.0. We no longer need to strip it -- just remove the REST mapper and GVR/Namespaced fields as before.
- **Adaptation 7 (Iterator Identity)**: Same as v0.8.x -- merge `MetadataNamePath` and `MetadataNamespacePath` cases.
- **Adaptation 8 (normalizeNamespaces)**: Now in `node_resolve.go` instead of `node.go`. Removal is split across two files.
- **Adaptation 9 (Validation)**: `validateResourceGraphDefinition` is removed; `validateResourceIDs` now directly includes the forEach validation that was previously in the wrapper. Takes `RGDConfig` parameter.
- **Resolver split**: In v0.8.x, our combined resolver factories were appended to `resolver.go`. Now `resolver.go` is IDENTICAL to upstream, and our additions live in a separate `combined_resolver.go` file.
- **Test utilities**: New `BuildTestXRSchema` function and `simpleSchemaToSpecWithTypes` helper in `testutil/generator/`. `discovery.go` reformatted for compactness with `stringMapSchema` helper.

## Adaptation 3: Replace `buildInstanceNode` with `buildInstanceNodeFromSchema` (`kro/graph/builder.go`)

### What to look for

Upstream splits instance building into multiple steps:
1. `buildInstanceSpecSchema()` -- builds spec from SimpleSchema
2. `crd.SynthesizeCRD()` -- creates the CRD
3. `buildStatusSchema()` -- infers status types from CEL + generates OpenAPI
4. `crd.SetCRDStatus()` -- updates CRD with inferred status
5. `buildInstanceNode()` -- creates the node from pre-computed components

### What to do

Replace ALL of steps 1-5 with a single `buildInstanceNodeFromSchema` function. This function:
- Parses status CEL expressions from `rgd.Status.Raw`
- Validates them using `inspectExpressionRestricted` (from `buildContext`)
- Compiles them using `bc.compile` (from `buildContext`)
- Builds the instance node with dependencies

Delete `buildInstanceSpecSchema()`, `buildStatusSchema()`, and `buildInstanceNode()` if they still exist.

**If upstream refactored these into different helper functions:** The key insight is that we don't need ANY of the SimpleSchema/CRD generation logic. We only need status CEL expression parsing, validation, compilation, and dependency extraction.

### Function-kro code (complete replacement)

```go
// buildInstanceNodeFromSchema builds the instance node from the ResourceGraph's
// status CEL expressions. In function-kro, we don't generate a CRD — Crossplane
// manages the XR CRD. We only need to parse status CEL expressions, validate
// them, compile them, and build the instance node.
func buildInstanceNodeFromSchema(
	bc *buildContext,
	rgd *v1beta1.ResourceGraph,
	nodeNames []string,
	inspector *ast.Inspector,
) (*Node, error) {
	// Parse the status template from the ResourceGraph.
	unstructuredStatus := map[string]interface{}{}
	if len(rgd.Status.Raw) > 0 {
		err := yaml.UnmarshalStrict(rgd.Status.Raw, &unstructuredStatus)
		if err != nil {
			return nil, fmt.Errorf("failed to unmarshal status schema: %w", err)
		}
	}

	// Extract CEL expressions from the status field.
	fieldDescriptors, noExpressionFields, err := parser.ParseSchemalessResource(unstructuredStatus)
	if err != nil {
		return nil, fmt.Errorf("failed to extract CEL expressions from status: %w", err)
	}

	if len(noExpressionFields) > 0 {
		return nil, fmt.Errorf("status fields without expressions are not supported: %v", noExpressionFields)
	}

	// Instance status expressions can ONLY reference resources, not schema.
	// Verify status expressions don't reference schema and populate References.
	for _, fieldDescriptor := range fieldDescriptors {
		expression := fieldDescriptor.Expression
		result, err := inspectExpressionRestricted(inspector, expression.Original, nodeNames)
		if err != nil {
			return nil, fmt.Errorf("status field %q expression %q: %w", fieldDescriptor.Path, expression.UserExpression(), err)
		}
		for _, dep := range result.ResourceDependencies {
			if !slices.Contains(expression.References, dep.ID) {
				expression.References = append(expression.References, dep.ID)
			}
		}
	}

	// Compile status expressions.
	for _, fd := range fieldDescriptors {
		if _, err := bc.compile(bc.env, fd.Expression); err != nil {
			return nil, fmt.Errorf("failed to compile status expression %q at path %q: %w", fd.Expression.UserExpression(), fd.Path, err)
		}
	}

	// Build instance node with status variables.
	var instanceDeps []string
	instanceStatusVariables := []*variable.ResourceField{}
	for _, statusVariable := range fieldDescriptors {
		path := "status." + statusVariable.Path
		statusVariable.Path = path

		deps, _, err := extractDependencies(inspector, statusVariable.Expression, nil)
		if err != nil {
			return nil, fmt.Errorf("failed to extract dependencies from expression %q: %w", statusVariable.Expression, err)
		}
		instanceDeps = append(instanceDeps, deps...)

		instanceStatusVariables = append(instanceStatusVariables, &variable.ResourceField{
			FieldDescriptor: statusVariable,
			Kind:            variable.ResourceVariableKindDynamic,
			Dependencies:    deps,
		})
	}

	instance := &Node{
		Meta: NodeMeta{
			ID:           InstanceNodeID,
			Type:         NodeTypeInstance,
			Dependencies: instanceDeps,
		},
		Template: &unstructured.Unstructured{
			Object: map[string]interface{}{
				"status": unstructuredStatus,
			},
		},
		Variables: instanceStatusVariables,
	}

	return instance, nil
}
```

- `buildStatusSchema()` -- CEL type inference for status schema generation

### Function-kro code (complete replacement)

```go
// getSchemaWithoutStatus creates a copy of the given schema with the status
// property removed. In function-kro, the schema comes from Crossplane (not a CRD),
// so we deep-copy it and strip the status field.
func getSchemaWithoutStatus(s *spec.Schema) (*spec.Schema, error) {
	if s == nil {
		return nil, nil
	}
	schemaCopy, err := kroschema.DeepCopySchema(s)
	if err != nil {
		return nil, fmt.Errorf("failed to deep copy schema: %w", err)
	}
	delete(schemaCopy.Properties, "status")
	return schemaCopy, nil
}
```

## Adaptation 9: Validation Changes (`kro/graph/validation.go`)

### What to look for

Upstream has two validation entry points:
- `validateResourceGraphDefinition()` -- validates Kind name, calls `validateResourceIDs()`, then validates forEach dimensions
- `validateResourceIDs()` -- validates individual resource IDs

Both take `*v1alpha1.ResourceGraphDefinition`.

Also `validateTemplateConstraints()` uses the REST mapper for namespace validation, and `validateIdentityFields()` reads `node.Meta.Namespaced`.

### What to do

1. **Merge validation functions:** Delete `validateResourceGraphDefinition()` and fold the forEach validation into `validateResourceIDs()`. Skip the Kind name validation (Crossplane manages the XR CRD).
2. **Change all type references** per the Type Mapping table.
3. **Change field accesses** per the Field Access Mapping table (`rgd.Spec.Resources` -> `rgd.Resources`).
4. **Keep `validateTemplateConstraints()`** but note it is never called from `buildRGResource` (the REST mapper call that fed it is removed). It still exists in the codebase but is dead code from our perspective.
5. **Fix `validateIdentityFields()`**: Replace `node.Meta.Namespaced` with `true` (always assume namespaced).

**If upstream renamed these functions:** Apply the same type changes to the new names.
**If upstream added new validation that uses REST mapper:** Remove that validation.

### Function-kro `validateResourceIDs`

```go
func validateResourceIDs(rgd *v1beta1.ResourceGraph, rgdConfig RGDConfig) error {
	seen := make(map[string]struct{})
	for _, res := range rgd.Resources {  // NOTE: rgd.Resources, not rgd.Spec.Resources
		if isKROReservedWord(res.ID) {
			return fmt.Errorf("%s: id %s is a reserved keyword in KRO", ErrNamingConvention, res.ID)
		}
		if !isValidResourceID(res.ID) {
			return fmt.Errorf("%s: id %s is not a valid KRO resource id: must be lower camelCase", ErrNamingConvention, res.ID)
		}
		if _, ok := seen[res.ID]; ok {
			return fmt.Errorf("%s: found duplicate resource IDs %s", ErrNamingConvention, res.ID)
		}
		seen[res.ID] = struct{}{}
	}

	// Validate forEach iterators after collecting all resource IDs
	resourceIDs := sets.NewString()
	for _, res := range rgd.Resources {
		resourceIDs.Insert(res.ID)
	}
	for _, res := range rgd.Resources {
		if err := validateForEachDimensions(res, resourceIDs, rgdConfig); err != nil {
			return err
		}
	}

	return nil
}
```

### `kro/graph/schema/resolver/crd_resolver.go`

Resolves schemas by extracting them from CRD objects (fallback for older Crossplane).

```go
package resolver

import (
	"fmt"

	extv1 "k8s.io/apiextensions-apiserver/pkg/apis/apiextensions/v1"
	"k8s.io/apimachinery/pkg/runtime/schema"
	"k8s.io/kube-openapi/pkg/validation/spec"

	kroschema "github.com/crossplane-contrib/function-kro/kro/graph/schema"
)

type CRDSchemaResolver struct {
	schemas map[schema.GroupVersionKind]*spec.Schema
}

func NewCRDSchemaResolver(crds []*extv1.CustomResourceDefinition) (*CRDSchemaResolver, error) {
	schemas := make(map[schema.GroupVersionKind]*spec.Schema)
	for _, crd := range crds {
		if crd == nil { continue }
		group := crd.Spec.Group
		kind := crd.Spec.Names.Kind
		for _, version := range crd.Spec.Versions {
			if version.Schema == nil || version.Schema.OpenAPIV3Schema == nil { continue }
			gvk := schema.GroupVersionKind{Group: group, Version: version.Name, Kind: kind}
			specSchema, err := kroschema.ConvertJSONSchemaPropsToSpecSchema(version.Schema.OpenAPIV3Schema)
			if err != nil {
				return nil, fmt.Errorf("failed to convert schema for %s: %w", gvk, err)
			}
			if specSchema.Properties == nil {
				specSchema.Properties = make(map[string]spec.Schema)
			}
			specSchema.Properties["metadata"] = kroschema.ObjectMetaSchema
			schemas[gvk] = specSchema
		}
	}
	return &CRDSchemaResolver{schemas: schemas}, nil
}

func (r *CRDSchemaResolver) ResolveSchema(gvk schema.GroupVersionKind) (*spec.Schema, error) {
	return r.schemas[gvk], nil
}
```

### `kro/graph/schema/resolver/combined_resolver.go`

Combined resolver factories that pair Crossplane-provided schemas with core Kubernetes type resolution. In v0.8.x these were appended to `resolver.go`; in v0.9.0 they live in a separate file since `resolver.go` is now identical to upstream.

```go
package resolver

import (
	"k8s.io/apiextensions-apiserver/pkg/generated/openapi"
	"k8s.io/apimachinery/pkg/runtime/schema"
	"k8s.io/apiserver/pkg/cel/openapi/resolver"
	"k8s.io/client-go/kubernetes/scheme"
	"k8s.io/kube-openapi/pkg/validation/spec"
)

func NewCombinedResolverFromSchemas(schemaMapResolver *SchemaMapResolver) resolver.SchemaResolver {
	return &combinedResolver{primary: schemaMapResolver, fallback: newCoreResolver()}
}

func NewCombinedResolverFromCRDs(crdResolver *CRDSchemaResolver) resolver.SchemaResolver {
	return &combinedResolver{primary: crdResolver, fallback: newCoreResolver()}
}

func newCoreResolver() resolver.SchemaResolver {
	return resolver.NewDefinitionsSchemaResolver(openapi.GetOpenAPIDefinitions, scheme.Scheme)
}

type combinedResolver struct {
	primary  resolver.SchemaResolver
	fallback resolver.SchemaResolver
}

func (c *combinedResolver) ResolveSchema(gvk schema.GroupVersionKind) (*spec.Schema, error) {
	s, err := c.primary.ResolveSchema(gvk)
	if err != nil { return nil, err }
	if s != nil { return s, nil }
	return c.fallback.ResolveSchema(gvk)
}
```

// DeepCopySchema creates a deep copy of a spec.Schema by marshaling and
// unmarshaling through JSON.
func DeepCopySchema(schema *spec.Schema) (*spec.Schema, error) {
	data, err := json.Marshal(schema)
	if err != nil {
		return nil, fmt.Errorf("failed to marshal schema: %w", err)
	}
	var copy spec.Schema
	if err := json.Unmarshal(data, &copy); err != nil {
		return nil, fmt.Errorf("failed to unmarshal schema: %w", err)
	}
	return &copy, nil
}

## Schema Resolution Architecture

Function-kro uses capability-based detection in `fn.go` to choose a schema resolution path. This is NOT in the KRO libraries -- it's in our `fn.go` entry point. But it's documented here because it explains why the adaptations above exist.

### Path 1: Required Schemas (Crossplane v2.2+)
```
HasCapability(REQUIRED_SCHEMAS) -> response.RequireSchema() -> request.GetRequiredSchemas()
    -> structToSpecSchema() -> SchemaMapResolver -> NewCombinedResolverFromSchemas()
    -> resolver.SchemaResolver passed to NewBuilder()
```

### Path 2: CRDs (Fallback for older Crossplane)
```
!HasCapability(REQUIRED_SCHEMAS) -> request CRDs via required_resources
    -> request.GetRequiredResources() -> NewCRDSchemaResolver()
    -> NewCombinedResolverFromCRDs() -> resolver.SchemaResolver passed to NewBuilder()
```

Both paths produce the same `resolver.SchemaResolver` interface that the graph builder consumes.

## Known Limitations

1. **No cluster-scope validation.** Upstream validates that cluster-scoped resources don't set a namespace (via REST mapper). Function-kro can't perform this check. Users setting namespace on cluster-scoped resources will get an error from Crossplane/the provider rather than from the function.

## Upstream Behavior We Intentionally Don't Need

1. **Namespace auto-defaulting (removed).** Upstream auto-defaults namespace from the instance for namespaced resources. We removed this (`normalizeNamespaces` in `runtime/node_resolve.go`) because Crossplane automatically sets the namespace on composed resources at the framework level. Composition authors don't need to set namespace in their resource templates.

2. **CRD synthesis (excluded).** Upstream generates CRDs from SimpleSchema definitions. Function-kro doesn't generate CRDs -- Crossplane manages the XR CRD lifecycle.

3. **Revision tracking (excluded).** Upstream tracks graph revisions with hashing and a registry. Function-kro is stateless per invocation -- no revision tracking needed.
